## 1. 测试设计

### 1.1 测试场景

模拟真实业务场景：应用程序持续向 MongoDB 副本集读写数据，在运行过程中触发主从切换，观察：

- ✅ 切换前：请求是否全部打到 AWS(27017)
- ✅ 切换过程：是否有请求失败、失败持续多久
- ✅ 切换后：请求是否自动切换到腾讯云(27018)
- ✅ 数据一致性：切换前后的数据是否完整

### 2.3 为什么对照组也没失败？

这得益于 MongoDB `rs.stepDown()` 的精妙设计：

```
rs.stepDown(60, 30) 内部流程:

1. 等待从节点 oplog 追上（最多 30 秒）
   ↓ 此时主节点仍在服务读写
2. 确认从节点追上后，才开始降级
   ↓
3. 主节点关闭已建立的"可写"连接
   客户端 driver 感知到 PRIMARY 失效
   ↓
4. 触发选举（通常 < 1 秒）
   ↓
5. 新 PRIMARY 产生
   ↓
6. driver 自动发现新 PRIMARY
   新连接建立，恢复写入

整个过程中：
- 请求间隔 100ms 的业务，会"错过"无主节点窗口
- driver 的 serverSelectionTimeoutMS=2000ms 也起到缓冲作用
- 新请求来时 PRIMARY 已经切换完成
```

| 阶段 | 耗时 | 说明 |
|------|------|------|
| **步骤1：提升腾讯云 priority** | ~0.3s | rs.reconfig() 执行 |

| **切换总耗时** | **~1.1 秒** | 从命令发起到业务恢复 |
| **业务"无请求"窗口** | **~0.5 秒** | 实际请求空档期 |

## 4. 核心验证点

### ✅ 验证点 1：请求自动路由到 PRIMARY

- 切换前 474 次请求 100% 打到 AWS(27017)
- 切换后 1,251 次请求 100% 打到腾讯云(27018)
- **driver 完全透明地完成了切换，应用代码无需修改**

### ✅ 验证点 2：切换速度

- 从 `stepDown` 执行到新 PRIMARY 可写：**~1.1 秒**
- 从业务视角看，只是 "一次请求可能稍慢"
- 符合生产级切换预期（通常 10-12 秒内）

### ✅ 验证点 3：数据零丢失

- `stepDown` 会等待从节点追平 oplog
- 切换过程中没有数据写入到"孤岛"
- 切换后腾讯云节点数据与 AWS 完全一致

### ✅ 验证点 4：Driver 自动容错

- **启用 retryWrites（推荐）**：切换期间的失败请求被自动重试，业务完全无感知
- **关闭 retryWrites**：由于请求间隔 + driver 缓冲，实测也无失败

## 5. 生产环境建议

基于本次 Demo 验证，生产环境推荐配置：

### 5.1 应用层配置

```python
# Python (pymongo)
MongoClient(
    "mongodb://aws-1:27017,aws-2:27017,tc-1:27017/?replicaSet=rs0"
    "&retryWrites=true"               # ★ 必须开启
    "&retryReads=true"                # ★ 必须开启
    "&w=majority"                     # ★ 多数派确认
    "&serverSelectionTimeoutMS=30000" # 连接超时 30s
)
```

```javascript
// Node.js
new MongoClient(uri, {
    retryWrites: true,
    retryReads: true,
    w: "majority",
    serverSelectionTimeoutMS: 30000
})
```

### 5.2 切换执行步骤（生产版）

```bash
# ============ 前置检查 ============
# 确认从节点追上
mongosh --host aws-1 --eval "rs.printSecondaryReplicationInfo()"
# 确认同步延迟 < 1s

# ============ 执行切换 ============
# 1. 提升腾讯云节点权重
mongosh --host aws-1 --eval '
  var conf = rs.conf();
  conf.members.find(m => m.host.includes("tc-1")).priority = 10;
  conf.members.find(m => m.host.includes("tc-1")).votes = 1;
  conf.members.find(m => m.host.includes("aws-1")).priority = 1;
  conf.version++;
  rs.reconfig(conf);
'

# 2. [可选] 业务层开启只读模式 10 秒
#    适用于金融级业务，进一步确保零丢失

# 3. 触发切换
mongosh --host aws-1 --eval 'rs.stepDown(60, 30)'

# 4. 验证切换
mongosh --host tc-1 --eval 'rs.status()'

# 5. [可选] 关闭业务只读模式
```

### 5.3 监控指标

切换过程中应监控：

- 应用错误率（预期：< 0.1%）
- 应用 P99 延迟（预期：临时增加 100-500ms）
- MongoDB 副本集状态（`rs.status()`）
- 同步延迟（`rs.printSecondaryReplicationInfo()`）

## 6. 风险与对策

| 风险 | 概率 | 应对 |
|------|------|------|
| 切换瞬间写失败 | 低 | 启用 retryWrites |
| 从节点未追上就切换 | 极低 | `stepDown` 会自动等待 |
| 应用 driver 不支持故障转移 | 视语言 | 使用官方 driver 最新版 |
| 跨云网络延迟导致切换慢 | 中 | 使用专线，延迟 < 50ms |
| 切换失败（无节点可选） | 极低 | 检查 votes 配置，保证有多数派 |

## 7. 回滚验证

如果切换后发现问题，可通过反向操作快速回滚（已验证）：

```javascript
// 在新主节点（腾讯云）执行
var conf = rs.conf();
conf.members.find(m => m.host.includes("aws-1")).priority = 10;
conf.members.find(m => m.host.includes("tc-1")).priority = 1;
conf.version++;
rs.reconfig(conf);

// 触发反向切换
rs.stepDown(60, 30);
```

**回滚耗时**：约 **1-2 秒**
